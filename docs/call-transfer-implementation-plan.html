1. Telnyx documentation findings

1.1 Blind transfer

Mechanism: Transfer call — POST /v2/calls/{call_control_id}/actions/transfer 

POST /v2/calls/{call_control_id}/actions/transfer
{ "to": "+E164 or sip:...", "from": "...", ... }

Expected webhooks: call.initiated → call.bridged (to Leg B) → call.answered or call.hangup

  • Commands are scoped to a call_control_id (one controllable leg).
  • If transfer fails, the original leg stays active and can receive other commands.
  • Supports audio_url, timeout_secs, from / from_display_name, park_after_unbridge, etc.
  • This is not SIP REFER — it is a Call Control command.

1.2 Attended (warm) transfer

Telnyx does not expose a single “warm transfer” primitive. Official patterns:

PatternTelnyx mechanismSource
Cold / blind actions/transfer Sending commands
Private warm Dial + Bridge composition AI handoff guide
3-way / supervisor Conference (join, hold, unhold, supervisor roles) Conferencing demo
Contact-center style park_after_unbridge: self on bridge/transfer Bridge calls

Documented attended sequence:

  1. Hold original caller
  2. Dial transfer destination
  3. Announce (optional)
  4. Complete (bridge or transfer)
  5. Hang up agent leg

1.3 Conference calling

Mechanism: Conference Commands

  • POST /v2/conferences — create from an existing call_control_id (call auto-joins)
  • POST /v2/conferences/{id}/actions/join — add legs
  • hold / unhold / mute / leave per participant
  • supervisor_role: barge, monitor, whisper
  • Expires when empty or after 4 hours
Hold is conference-scoped. There is no documented per-call actions/hold on a bare leg. Caller hold during warm transfer requires conference join + conference hold, or bridge options like hold_after_unbridge.

1.4 SIP REFER support

Three distinct surfaces — do not conflate:

SurfaceAPIWebhooksUse case
Call Control Refer POST /calls/{id}/actions/refer + sip_address call.refer.started, .completed, .failed SIP Refer a call
Transfer command POST /calls/{id}/actions/transfer + to call.initiated, call.bridged, … PSTN/SIP URI dial-out transfer
SIP Trunking REFER SIP-level REFER + Diversion header Trunk validation rules External transfers
TeXML Refer <Refer><Sip>…</Sip></Refer> TeXML callbacks TeXML apps only
Recommendation for VSP v1: use transfer + bridge + conference, not REFER. Reserve REFER for a later phase if external SIP handoff with REFER-specific semantics is required.

1.5 Bridge lifecycle

CommandExpected webhooks
Dial (POST /v2/calls)call.initiated, call.answered or call.hangup
Bridgecall.bridged on both legs
Transfercall.initiated, call.bridged to B, call.answered or call.hangup

Bridge options relevant to transfer:

  • park_after_unbridge: self — leg survives when bridge ends (parked, not hung up)
  • hold_after_unbridge: true — leg put on hold after unbridge
  • prevent_double_bridge: true — reject if target already bridged

Queueing doc: issuing bridge, transfer, conference join, or refer dequeues a call and changes its media path.

1.6 Call leg ownership

Telnyx model:

  • Each leg has its own call_control_id.
  • call_session_id may group related legs.
  • Dial creates a new leg with a new ID.
  • link_to + bridge_on_answer (used today) ties legs and auto-bridges on answer.

VSP model today (lib/telnyxCallControl.js):

POST /v2/calls

  bridge_on_answer: true

2. Current architecture vs requirements

ComponentTransfer readiness
lib/telnyxCallControl.jsdial, answer, hangup, record_start only — no transfer/bridge/conference/refer
lib/inboundCallControl.jsBridge-grace FSM (connecting → bridged); no transfer stages
lib/callControlSessionStore.jsInbound session + leg index; no transfer session
Web softphone v2Hold/mute via WebRTC SDK only; “Add Call” UI disabled
CallLogNo transfer metadata
RecordingStarted on PSTN leg after bridge — transfer must preserve or stop recording explicitly
Critical mismatch: Inbound PSTN ↔ WebRTC agent calls are Call-Control-bridged, not peer-only WebRTC. WebRTC call.hold() sets SDK held = sendonly on the agent leg only. That does not reliably hold the PSTN caller on the CC side. Warm transfer must be orchestrated on Call Control.

3. Recommended architecture

3.1 Design principles

  1. Backend-orchestrated Call Control — all transfer state in API + webhooks, not browser SDP tricks.
  2. Separate transfer session from inbound routing session — extend, don’t rewrite, inboundCallControl.js bridge-grace logic.
  3. Explicit leg registry per active call: callerLegId, agentLegId, optional consultLegId, conferenceId.
  4. Phase delivery — blind first, then attended, then conference/3-way.
  5. No REFER in v1 unless product explicitly requires external SIP REFER handoff.

3.2 High-level flow

Blind transfer

Agent (WebRTC) → POST /softphone/transfer/blind → VSP API VSP API → transfer(callerLegId, to=target) → Telnyx CC Telnyx CC → New outbound leg → Transfer Target Telnyx CC → call.bridged / call.answered → VSP API (webhooks) Telnyx CC → hangup → Agent WebRTC leg VSP API → transfer.completed → Agent UI

Attended transfer

Agent → POST /transfer/consult/start API → conference.create(callerLeg) + hold(caller) API → dial(consult) [CC-dialed consult leg — recommended] Agent → Consult with target (via CC bridge/conference) Agent → POST /transfer/consult/complete API → bridge/join caller to consult + hangup(agentLeg)

3.3 Transfer session (Redis)

New keys: cts:{transferId} (Call Transfer Session), TTL 1 hour.

type TransferSession = {
  transferId: string;
  tenantId: string;
  agentUserId: string;
  mode: 'blind' | 'attended';
  stage: 'idle' | 'holding' | 'consulting' | 'completing'
       | 'completed' | 'failed' | 'cancelled';
  callerLegId: string;       // PSTN or remote party
  agentLegId: string;        // WebRTC CC leg
  consultLegId?: string;
  conferenceId?: string;
  inboundSessionId?: string; // link to ccs:{inboundCallControlId}
  callSessionId: string;
  target: { type: 'pstn' | 'extension' | 'sip'; value: string };
  webrtcCallId?: string;
  createdAt: string;
};

  7. Recording + CDR after transfer
  8. Regression: inbound accept, bridge grace, voicemail fallback unchanged

11. Phased implementation plan

  • Phase 1 — Blind transfer (MVP)
    transferCall, bridgeCalls; callTransferControl.js; active-call index; POST /transfer/blind; webhook handling; frontend blind UI; optional CallLog metadata.
    Leg rule: transfer on callerLegId, not agent leg.
  • Phase 2 — Attended transfer
    Conference create/join/hold/unhold; consult start / complete / cancel APIs; frontend attended UX; recording cleanup.
  • Phase 3 — Conference / 3-way (optional)
    Supervisor roles; “Add Call” as conference join without completing transfer.
  • Phase 4 — SIP REFER (only if required)
    referCall + call.refer.* webhooks; external SIP policy. Not a substitute for Phase 1 transfer.
