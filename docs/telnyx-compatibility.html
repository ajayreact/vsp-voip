1. Executive Summary

Overall verdict: Largely compatible with documented Telnyx patterns, with intentional scope limits and a few operational gaps.

VSP-VOIP correctly uses Telnyx REST API v2 (https://api.telnyx.com/v2), Bearer authentication, Ed25519 webhook verification, Call Control commands, TeXML responses, Telephony Credentials for WebRTC, Messaging API, and Recordings API — all aligned with Telnyx documentation. Gaps are mainly missing product features (LNP porting API, 10DLC, MMS) and operational hardening (quota enforcement, dual inbound path complexity), not incorrect API usage.

Dual inbound path complexity: Numbers may be assigned to both TeXML connection (/webhook) and Call Control app (/webhook/call-control). Telnyx routes based on number assignment. WebRTC ring groups require Call Control; TeXML cannot dial WebRTC clients. Ensure production numbers use Call Control as primary inbound path.

Voice telemetry

lib/voiceTelemetry.js captures MOS, jitter, packet loss from call.hangup events and polls SIP registration status — aligned with Telnyx webhook payload fields and credential connection registration check.

Verification behavior

  • Compatible: Uses official telnyx SDK TelnyxWebhook with telnyx-signature-ed25519 and telnyx-timestamp headers.
  • Gap: If TELNYX_PUBLIC_KEY is unset, verification is skipped (warns in logs). Telnyx docs require verification in production.
  • Compatible: Returns 200 quickly on webhook handlers (async processing).
  • Compatible: Auto-configures webhook URLs on startup when API_PUBLIC_URL is set.
Production requirement: Always set TELNYX_PUBLIC_KEY in production. Telnyx docs state: "Webhooks are Ed25519-signed — verify signatures before trusting payloads."
